hashids

Generate short obfuscated strings from integers

available in JavaScript, Ruby, Python, Java, PHP, Perl, CoffeeScript, Objective-C, C++11, Go, Lua, Elixir, ColdFusion, Groovy and for Node.js & .NET

Uses

Hashids started from a need to have short unguessable ids.
md5 wasn't cutting it because it was too long, base64 was guessable. Hence, the hashids library was born.

Today it's used in url shorteners, keeping several integers together in single id, or simply as short unique identifiers.

Features

  1. Creates short unique ids. Duh.
  2. Allows custom alphabet as well as salt, so ids are unique only to you.
  3. Ids (as well as incremental integers) are unguessable.
  4. Code is about 300 lines, fast and does not depend on external libraries.

Compatibility table

Versions are tagged as <major>.<minor>.<patch>. Current stable version across implementations is 1.0:

  • Major version is incremented when interface functions/parameters change.
  • Minor version is incremented when the algorithm changes (ids change).
  • Patch version is incremented when individual implementation do minor adjustments.

As usual, check the CHANGELOG of each implementation. Since produced ids might differ on library update, be sure to include exact version of Hashids you want when using package managers, so ids don't change on you suddenly. For example for npm:

JSON"dependencies": {
  "hashids": "1.0.0"
}

And not: "hashids": "~1.0.0"

How does it work?

Hashids works similarly to the way integers are converted to hex, but with a few exceptions:

  1. The alphabet is not base16, but base base62 by default.
  2. The alphabet is also shuffled based on salt.

This is a JavaScript snippet that shows regular conversion of integers to hex (that's right, you don't see do...while that often). And if you actually look at the Hashids JavaScript implementation, this function is from there (but simplified for this example):

JavaScriptfunction toHex(input) {
  
  var hash = "",
    alphabet = "0123456789abcdef",
    alphabetLength = alphabet.length;
  
  do {
    hash = alphabet[input % alphabetLength] + hash;
    input = parseInt(input / alphabetLength, 10);
  } while (input);
  
  return hash;
  
}

So, if we do toHex(1234), we will get back 4d2 which is that integer's hex value. If we increase the alphabet to abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ1234567890, our output will now be shorter because there's more characters to work with now — so it becomes t5.

If we take it one step further and shuffle the alphabet before encoding, we will get a different value. But how could we shuffle the alphabet consistently, so that with every "shuffle string" the alphabet characters would be ordered the same every time? That's where Hashids uses a variation of Fisher-Yates algorithm:

JavaScriptfunction consistentShuffle(alphabet, salt) {
  
  var integer, j, temp, i, v, p;
  
  for (i = alphabet.length - 1, v = 0, p = 0; i > 0; i--, v++) {
  
    v %= salt.length;
    p += integer = salt[v].charCodeAt(0);
    j = (integer + v + p) % i;
    
    temp = alphabet[j];
    alphabet = alphabet.substr(0, j) + alphabet[i] + alphabet.substr(j + 1);
    alphabet = alphabet.substr(0, i) + temp + alphabet.substr(i + 1);
    
  }
  
  return alphabet;
  
}

It's fast and simple, and most importantly it does exactly what we need — shuffles the alphabet based on user's input (their salt value).

That way when user A passes the salt value abc1 our alphabet becomes cUpI6isqCa0brWZnJA8wNTzDHEtLXOYgh5fQm2uRj4deM91oB7FkSGKxvyVP3l

And when user B passes the salt value abc2, the alphabet becomes tRvkhHx0ZefcF46YuaAqGLDKgM1W5Vp2T8n9s7BSoCjiQOdrEbJmUINywzXP3l

You can see that the shuffle is pretty good even when salt value is not that much different. This is what makes the base of Hashids work. Now we are able to encode one integer based on the salt value we provide.

But Hashids is able to encode several integers into one id. This is done by reserving a few characters from the alphabet as separators. These characters are then not used in encoding, but instead do the job of purely separating the real encoded values.

Let's take an example input 1,2,3 and encode it with Hashids without salt: o2fXhV. The actual values in this output are highlighted: o2fXhV. Letters f and h are simply used as separators.

Letter o is reserved for another type of job - it acts as a "lottery" character to randomize incremental input. Currently if we encode numbers as we increment them, the output is somewhat predictable:

  • 1,2,3 = o2fXhV
  • 1,2,4 = o2fXhd
  • 1,2,5 = o2fXh6
  • 1,2,6 = o2fXhz
  • 1,2,7 = o2fXhR

So the lottery character is used to do another iteration of consistent shuffle, before starting the actual encoding so the numbers end up looking like this. A lot less predictable, with a disadvantage of having an extra character:

  • 1,2,3 = o2fXhV
  • 1,2,4 = pYfzhG
  • 1,2,5 = qxfOhN
  • 1,2,6 = rkfAhN
  • 1,2,7 = v2fWhy

This is a quick overview of how Hashids is structured. Decoding is done the same way but in reverse — of course in order for that to work, Hashids itself needs the salt value from you in order to decode ids correctly.

If you give Hashids the wrong salt value and it just so happens that it decodes back to some random integers, Hashids does a quick check by encoding those integers to be sure the initial id matches. If it does not, the integers are discarded.

This library was written with the intent of placing generated ids in visible places - like the URL. Therefore, the algorithm tries to avoid generating most common English curse words by never placing the following letters next to each other:

c,C,s,S,f,F,h,H,u,U,i,I,t,T

This is done by using them as separators (refer to how hashids works).

So, since letters like F and U cannot be next to each other in a generated id, the f-bomb is not possible (no matter what your salt value is).

Are there any alternatives?

Yes, there are a few and you're encouraged to pick the most appropriate. Hashids is not a perfect solution to everything.

  1. Base64 encode. This is the most straightforward approach — most languages have these functions. If you don't need all the fancy extras Hashids offers, this method will work just fine. It's faster too. Read more.
  2. Generate ids based on timestamp. If you can afford certain degree of collisions, you could compose an id that's built on the fly. Use a counter (if you have one) + timestamp (even better if in milliseconds) + some system value (either an IP address or some machine id) + a random integer. Many big companies implement this approach because it works well in distributed systems. These ids are generated independently of each other and the risk of collisions is so tiny it's negligible.

What not to do

  1. Do not encode strings. We've had several questions asking why there's no such feature — it seems so easy to add. We will not add this feature for security purposes, doing so encourages people to encode sensitive data, like passwords. This is the wrong tool for that.
  2. Do not encode sensitive data. This means passwords, emails, any kind of personal user data. This is not a true encryption algorithm. There are people that dedicate their lives to cryptography and there are plenty of more appropriate algorithms: bcrypt, md5, aes, sha1, blowfish. Here's a full list.

Collisions

There is no collisions because the method is based off of integer to hex conversion. As long as you don't change constructor arguments midway, the generated output will stay unique to your salt.

You can also test this by generating a billion ids and checking for collisions. Be sure to account for your alphabet, since default Hashids alphabet is case-sensitive (aaa is a different id than AAA).

How to decode

There is no way to decode the output without knowing the salt (refer to how hashids works). The only known way is to use a brute-force attack. However, since the integers behind ids are of so little value to the attacker, there's not much reason to attack them in the first place.

If you've found a bug, please open a github issue in the appropriate repository. Bonus points if you submit a pull request with it.
